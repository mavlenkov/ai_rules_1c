# Agents Specification

## Purpose
Предоставить систему специализированных AI-ассистентов для разных этапов разработки на 1С.

## Requirements

### Requirement: Специализация по ролям
Каждый агент должен быть оптимизирован под конкретную роль в процессе разработки.

#### Scenario: Разработчик выбирает агента для написания кода
- GIVEN нужно реализовать новую функциональность
- WHEN разработчик вызывает агента `1c-developer`
- THEN агент использует MCP-инструменты, следует правилам кодирования и проводит самопроверку

#### Scenario: Нужно спроектировать архитектуру решения
- GIVEN требуется принять архитектурное решение
- WHEN разработчик вызывает агента `1c-architect`
- THEN агент анализирует паттерны, предлагает варианты и создаёт диаграммы

### Requirement: Группировка агентов по областям
Агенты объединены в группы по функциональным областям.

#### Scenario: Группа "Разработка"
- GIVEN задача разработки
- WHEN выбран один из агентов: developer, architect, analytic
- THEN агент создаёт код, архитектуру или спецификацию

#### Scenario: Группа "Качество"
- GIVEN задача проверки качества
- WHEN выбран один из агентов: code-reviewer, arch-reviewer, error-fixer, refactoring, performance-optimizer
- THEN агент проверяет код, исправляет ошибки или оптимизирует

#### Scenario: Группа "Документация и планирование"
- GIVEN задача документирования или планирования
- WHEN выбран один из агентов: doc-writer, planner, tester
- THEN агент создаёт документацию, план задач или тесты

#### Scenario: Группа "Метаданные"
- GIVEN задача работы с метаданными 1С
- WHEN выбран агент metadata-manager
- THEN агент создаёт, редактирует, валидирует объекты конфигурации

### Requirement: SDD-интеграция
Все агенты должны поддерживать работу с OpenSpec и другими SDD-фреймворками.

#### Scenario: Агент обнаруживает OpenSpec в проекте
- GIVEN в проекте есть папка `openspec/specs/`
- WHEN любой агент начинает работу
- THEN он читает `rules/sdd-integrations.mdc` и следует workflow OpenSpec

### Requirement: Workflow между агентами
Агенты должны дополнять друг друга в рамках единого процесса.

#### Scenario: Последовательный workflow
- GIVEN новая большая фича
- WHEN агенты вызываются последовательно: planner → architect → developer → code-reviewer → tester
- THEN каждый агент использует результаты предыдущего

## Agent List

| Агент | Роль | Основная функция |
|-------|------|------------------|
| **developer** | Разработка | Написание кода с MCP-инструментами и самопроверкой |
| **architect** | Разработка | Проектирование архитектуры, trade-off analysis |
| **analytic** | Разработка | Бизнес-анализ, PRD, ТЗ, спецификации |
| **code-reviewer** | Качество | Ревью кода с confidence scoring |
| **arch-reviewer** | Качество | Ревью архитектурных решений |
| **error-fixer** | Качество | Исправление ошибок и BSL warnings |
| **refactoring** | Качество | Рефакторинг с сохранением функциональности |
| **performance-optimizer** | Качество | Оптимизация производительности |
| **doc-writer** | Документация | Техническая документация для пользователей |
| **planner** | Планирование | Декомпозиция задач и планирование |
| **tester** | Тестирование | Разработка и выполнение тестов |
| **metadata-manager** | Метаданные | Создание/редактирование объектов конфигурации |
