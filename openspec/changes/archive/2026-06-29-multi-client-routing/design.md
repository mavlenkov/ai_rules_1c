## Context

Форк раскладывает один набор субагентов (`content/agents/*.md`) под несколько AI-клиентов. Каждый агент объявляет абстрактный `modelTier` (`reasoning`/`coding`/`light`); установщик подставляет конкретную модель из `SUBAGENT_MODEL_<TIER>` (`.dev.env`). Параметр **один на проект** и подставляется во всех клиентов одинаково.

Эмпирически (OpenCode 1.17.11) подтверждено:
- OpenCode принимает только `provider/model` (`zai-coding-plan/glm-5.2`, `deepseek/deepseek-v4-pro`); голого `sonnet`/`opus`/`haiku` в списке моделей нет → битое имя молча откатывается на дефолт.
- Claude Code принимает короткие алиасы (`sonnet`).
- При последней перекатке 7 проектов `model: sonnet` записан во всех клиентов, т.е. opencode-агенты сейчас с битым именем.

Фокус — два реально используемых клиента: **Claude Code** (только Anthropic) и **OpenCode** (Z.AI Coding Plan + DeepSeek). **Codex** — не установочный таргет, а внешний критик, вызываемый изнутри обоих (его модель живёт в его собственном конфиге, в развязку моделей не входит).

Точки внедрения уже tool-aware: `install.ps1 → Invoke-PlacePhase` итерирует `foreach ($tool in $ActiveTools)`; `scripts/install.sh` — `for tool, adapter in adapters.items()`. `tool` известен ровно там, где резолвится модель.

Связанное требование пользователя: внешний критик (Codex) должен срабатывать **автоматически по правилам**, а при недоступности Codex — деградировать на имеющиеся модели.

## Goals / Non-Goals

**Goals:**
- Разрешать модель субагента **per-client**: дефолт + override по клиенту, в обоих установщиках одинаково.
- Не записывать заведомо битые имена (алиас в клиент, требующий `provider/model`).
- Сохранить обратную совместимость: общий `SUBAGENT_MODEL_<TIER>` без суффикса продолжает работать.
- Ввести автоматический внешний критик в quality gate с graceful fallback и корректным контекстом правил.

**Non-Goals:**
- Управление моделью Codex (она в его собственном конфиге).
- Поддержка DeepSeek в Claude Code через прокси (отдельная тема, не здесь).
- Превращение Codex в установочный таргет.
- Изменение самого набора ярусов (остаются `reasoning`/`coding`/`light`).

## Decisions

### A1. Формат per-client override — суффикс `__<TOOL>`

Имя ключа: `SUBAGENT_MODEL_<TIER>__<TOOL>`, где `<TOOL>` = id адаптера в верхнем регистре с заменой дефиса на подчёркивание: `claude-code → CLAUDE_CODE`, `opencode → OPENCODE`, `cursor → CURSOR`, `kilocode → KILOCODE`, `codex → CODEX`. Пример: `SUBAGENT_MODEL_CODING__OPENCODE=zai-coding-plan/glm-5.2`.

**Альтернативы:** (1) префикс `OPENCODE_SUBAGENT_MODEL_CODING` — хуже группируется визуально и при `grep SUBAGENT_MODEL`; (2) вложенные секции/неймспейс — `.dev.env` плоский (`KEY=VALUE`), вложенность чужеродна; (3) JSON-значение с per-client мапой — ломает простой парсинг grep'ом (toolkit-команды парсят `.dev.env` через grep, не через `source`). Двойное подчёркивание `__` выбрано как разделитель, не конфликтующий с именами ярусов/клиентов.

### A2. Каскад резолюции

`resolve(tier, tool) = value(SUBAGENT_MODEL_<TIER>__<TOOL>) ?? value(SUBAGENT_MODEL_<TIER>) ?? ""`. Пусто → поле `modelHint`/`model` не эмитится, клиент берёт свою модель по умолчанию. Каскад одинаков в `install.ps1` и `install.sh`.

**Альтернатива «Claude-default» (дефолт только для Claude-подобных клиентов)** рассмотрена и отклонена как частный случай: проверка формата (A3) и так гасит битый алиас для opencode/kilo, а симметричный каскад проще и предсказуемее.

### A3. Проверка формата имени по клиенту

Для клиентов, требующих `provider/model` (`opencode`, `kilocode`), если итоговое имя **не содержит `/`**, модель НЕ подставляется (поле опускается) и печатается WARNING вида: `SUBAGENT_MODEL_CODING='sonnet' не похоже на provider/model — пропущено для opencode; задай SUBAGENT_MODEL_CODING__OPENCODE`. Это превращает текущий молчаливый баг в явный сигнал и даёт корректный дефолт (модель клиента) вместо битого имени. Для `claude-code`/`cursor` проверка не применяется (короткие алиасы валидны).

**Список клиентов, требующих `provider/model`,** держим явным в обоих установщиках (`opencode`, `kilocode`), чтобы не угадывать эвристикой.

### A4. Починка уже битых конфигов

Отдельной миграционной логики в установщике не добавляем — корректный результат даёт обычная перекатка после фикса: при пустом/несовместимом значении поле `model` опускается, и битый `model: sonnet` исчезает из opencode-агентов. Для развёрнутых проектов — добавить `SUBAGENT_MODEL_CODING__OPENCODE` (и при желании reasoning/light) в их `.dev.env` и перекатать.

### B1. Триггер автоматического критика

Внешний критик запускается на этапе **verification gate** (soft-gate в `verification-checklist.md`) для **нетривиального** кода — той же границы, что отделяет full-cycle от quick-fix в Triage (`AGENTS.md → Development Procedure`). Quick-fix и docs-fix критика не требуют. Это не добавляет новый этап, а усиливает существующий «user-explicit code review» до автоматического.

### B2. Per-client детекция доступности Codex

Абстракция: «способ вызова Codex для текущего клиента доступен?». Конкретика per-client:
- **Claude Code** — codex-плагин (`codex:rescue`/`codex` CLI). Признак готовности — наличие `codex` в PATH (плагин использует тот же runtime).
- **OpenCode** — прямой `codex exec` через shell (`command -v codex`).

Правило формулирует проверку декларативно (через секции по клиенту), не привязываясь к одной реализации. Недоступность = любой из: бинарь не найден, ошибка запуска, таймаут.

### B3. Fallback-цепочка

```
verification gate (нетривиальный код):
  ① обязательные hard-валидаторы:  syntaxcheck → check_1c_code → review_1c_code(Напарник)
  ② внешний критик (авто):
       try   → Codex  (контекст правил + diff)
       catch → 1c-code-reviewer @ <модель текущего клиента>
```
`review_1c_code` (1С:Напарник) — **обязательный слой ①**, не часть fallback. Codex — дополнительный «свежий взгляд» в слое ②. Fallback-субагент `1c-code-reviewer` использует per-client модель из области A (Claude Code: `sonnet`; OpenCode: `glm-5.2`), т.е. B опирается на A.

**Альтернатива** «Codex обязателен» отклонена: ломает работу там, где Codex не настроен, и противоречит общему принципу graceful fallback (как у MCP-цепочек).

### B4. Контекст правил для Codex

Codex нативно читает только корневой `AGENTS.md`; on-demand правила (`content/rules/*`) он сам не загружает (это верно для всех клиентов — `alwaysApply:false`). Критичный файл для ревью — `anti-patterns.md` (это «code-review scoring rubric»). Поэтому при вызове Codex родитель **прикладывает**: (1) `anti-patterns.md` (рубрику), (2) индекс `coding-standards.md`, (3) diff/изменённые файлы. Это закрывает разрыв между «Codex видит только AGENTS.md» и «ревью должно идти по нашей 1С-рубрике».

### B5. Размещение

Новое on-demand правило `content/rules/external-review.md` (с per-client секциями вызова и fallback). Хук — короткая ссылка из `verification-checklist.md` (soft-gate) и упоминание в `AGENTS.md → Quality`. Правило раскладывается во все клиенты обычным механизмом.

### C1. Документирование биллинг-семантики ярусов

В `subagents.md`/`dev-standards-core.md` зафиксировать: ярус трактуется по биллингу клиента — per-token (Claude Code) → стоимость×способность; flat-подписка (OpenCode/Z.AI) → способность×латентность. Поэтому в OpenCode `reasoning` и `coding` допустимо указывать одной моделью (`glm-5.2`), а `light` — быстрой (`glm-5-turbo`). Это объясняет, почему per-client значения могут не повторять ярусную «лесенку» Claude Code.

## Risks / Trade-offs

- **Рассинхрон списка «provider/model-клиентов» с реальностью** (вдруг новый клиент тоже требует префикс) → список явный в обоих установщиках + покрыт доком; добавление нового клиента = одна правка.
- **Дрейф двух установщиков** (ps1/sh реализуют каскад по-разному) → один и тот же каскад и список клиентов, проверка ручным прогоном обоих на временной директории (тестов в репо нет).
- **Codex-критик ест лимиты/время** при автозапуске на каждом нетривиальном изменении → триггер только для нетривиального кода (B1) + fallback не блокирует, если Codex недоступен.
- **Free-tier/нестабильность провайдеров OpenCode** (напр. `opencode/deepseek-v4-flash-free`) → в примерах `.dev.env.example` используем оплаченные по подписке модели (`zai-coding-plan/*`), free-tier только упоминаем в комментарии.
- **Расширение расхождения с upstream** (суффиксы, 3-й ярус уже разошлись, теперь ещё критик) → фиксируем в `FORK-TODO.md` с пометкой «при merge upstream сохранять форк-значения».

## Migration Plan

1. Реализовать каскад + проверку формата в `install.sh` и `install.ps1` (фаза A).
2. Обновить `.dev.env.example` и доки (A + C1).
3. Реализовать правило `external-review.md` + хуки (фаза B).
4. Прогон обоих установщиков на временной директории: проверить per-client модели (claude-code=alias, opencode=provider/model), WARNING на алиасе для opencode, отсутствие `model` при пустом override.
5. Развёрнутые проекты: добавить `__OPENCODE`-модели в `.dev.env`, перекатать, проверить `opencode models`-валидность и отсутствие битого `sonnet`.
6. Обновить `FORK-TODO.md`.

**Rollback:** изменения аддитивны и tool-локальны; откат — вернуть прежние резолверы; общий `SUBAGENT_MODEL_<TIER>` продолжает работать, так что одноклиентные проекты не затрагиваются.
